# Frontend Guide

이 문서는 Brelio의 프론트엔드 개발 규칙입니다.
입문자가 작업하더라도 코드 스타일과 구조가 크게 흔들리지 않도록, 작게 만들고 명확하게 나누는 것을 목표로 합니다.

## 기본 원칙

- TypeScript로 작성합니다.
- React 함수 컴포넌트를 사용합니다.
- 한 번에 많은 기능을 만들지 않습니다.
- 화면, API 요청, 타입, 공통 UI를 역할별로 분리합니다.
- 복잡한 추상화보다 읽기 쉬운 코드를 우선합니다.
- 하드코딩된 서버 주소, 토큰, 비밀값을 넣지 않습니다.
- 이벤트 핸들러 이름은 `handle`로 시작합니다.

좋은 예:

```tsx
function ArticleCard() {
  const handleOpenArticle = () => {
    // ...
  };

  return <button onClick={handleOpenArticle}>Read</button>;
}
```

피해야 할 예:

```tsx
function ArticleCard() {
  const click = () => {
    // 이름만 봐서는 무엇을 하는지 알기 어렵습니다.
  };

  return <button onClick={click}>Read</button>;
}
```

## 권장 폴더 구조

프론트엔드 코드는 `client/src` 아래에 둡니다.

```txt
client/src/
  api/
    articleApi.ts
  components/
    ArticleCard.tsx
  pages/
    ArticleListPage.tsx
    ArticleDetailPage.tsx
  routes/
    AppRoutes.tsx
  types/
    article.ts
  App.tsx
  main.tsx
  styles.css
```

처음부터 모든 폴더를 만들 필요는 없습니다.
기능이 생길 때 필요한 폴더만 추가합니다.

## 파일 역할

- `pages`: 라우트에 직접 연결되는 화면입니다.
- `components`: 여러 화면에서 재사용하거나 화면 안에서 분리할 UI입니다.
- `api`: 서버와 통신하는 함수입니다.
- `types`: 여러 파일에서 공유하는 TypeScript 타입입니다.
- `routes`: React Router 설정입니다.

한 파일에 화면, API 요청, 타입, 복잡한 상태 로직을 모두 넣지 않습니다.

## 컴포넌트 규칙

- 컴포넌트 파일 이름은 PascalCase를 사용합니다.
- 컴포넌트는 가능하면 한 가지 역할만 맡습니다.
- props 타입은 컴포넌트 위에 선언합니다.
- props 이름은 구체적으로 작성합니다.
- UI가 100줄을 크게 넘고 역할이 섞이면 컴포넌트 분리를 검토합니다.

좋은 예:

```tsx
type ArticleCardProps = {
  title: string;
  summary: string;
  sourceName: string;
  onOpen: () => void;
};

function ArticleCard({ title, summary, sourceName, onOpen }: ArticleCardProps) {
  return (
    <article>
      <h2>{title}</h2>
      <p>{summary}</p>
      <span>{sourceName}</span>
      <button type="button" onClick={onOpen}>
        Open
      </button>
    </article>
  );
}

export default ArticleCard;
```

## TypeScript 규칙

- `any`는 사용하지 않습니다. 정말 필요하면 먼저 타입을 좁힐 방법을 찾습니다.
- 서버에서 받는 데이터 타입은 `types` 폴더에 둡니다.
- optional 값은 화면에 표시하기 전에 기본값이나 조건 처리를 합니다.
- API 응답 타입과 화면에서 쓰는 타입이 달라지면 변환 함수를 둡니다.

좋은 예:

```ts
export type Article = {
  id: string;
  title: string;
  summary: string;
  originalUrl: string;
  publishedAt: string;
};
```

## API 요청 규칙

- API 요청은 컴포넌트 안에 직접 길게 작성하지 않습니다.
- 서버 주소는 환경 변수로 관리합니다.
- API 함수 이름은 동작이 드러나게 작성합니다.
- 비동기 코드는 `async/await`을 사용합니다.

예시:

```ts
const API_BASE_URL = import.meta.env.VITE_API_BASE_URL;

export async function getArticles() {
  const response = await fetch(`${API_BASE_URL}/articles`);

  if (!response.ok) {
    throw new Error("Failed to fetch articles");
  }

  return response.json();
}
```

## 상태 관리 규칙

- 한 컴포넌트 안에서만 쓰는 값은 `useState`를 사용합니다.
- 서버에서 가져오는 데이터는 TanStack Query 도입 후 query로 관리합니다.
- 여러 화면에서 공유해야 하는 값이 생기기 전까지 전역 상태를 만들지 않습니다.
- 상태 이름은 화면 의미가 드러나게 작성합니다.

좋은 예:

```tsx
const [selectedArticleId, setSelectedArticleId] = useState<string | null>(null);
```

피해야 할 예:

```tsx
const [data, setData] = useState(null);
```

## 스타일 규칙

- Tailwind CSS를 기본 스타일 도구로 사용합니다.
- 같은 스타일 조합이 반복되면 컴포넌트로 분리합니다.
- 색상과 간격은 일관되게 사용합니다.
- 클릭 가능한 요소는 `button` 또는 `a`처럼 의미에 맞는 HTML 태그를 사용합니다.
- 모바일 화면에서 깨지지 않는지 확인합니다.

## 라우팅 규칙

- 페이지 단위 컴포넌트는 `pages` 폴더에 둡니다.
- 라우트 경로는 짧고 명확하게 작성합니다.

권장 경로:

```txt
/articles
/articles/:articleId
/notes
/daily
```

## 에러와 로딩 처리

서버 데이터를 불러오는 화면에는 아래 세 가지 상태를 고려합니다.

- 로딩 중
- 에러 발생
- 데이터 없음

데이터가 없을 때 빈 화면만 보여주지 않습니다.
사용자가 무엇을 보고 있는지 알 수 있는 문구를 표시합니다.

## 접근성 기본 규칙

- 버튼에는 클릭 동작이 명확한 텍스트를 넣습니다.
- 이미지를 사용할 경우 의미 있는 이미지에는 `alt`를 작성합니다.
- 링크는 실제 이동에만 사용하고, 버튼은 화면 안 동작에 사용합니다.
- 키보드로 이동해도 주요 기능을 사용할 수 있어야 합니다.

## 프론트엔드 개발 로드맵

프론트엔드는 아래 순서대로 개발합니다.
각 단계는 화면에서 직접 확인할 수 있는 작은 결과물을 목표로 합니다.
앞 단계가 안정되기 전에는 다음 단계로 넘어가지 않습니다.

### 1. 기본 화면 뼈대

목표: 앱을 열었을 때 사용자가 어떤 서비스인지 바로 알 수 있게 만듭니다.

만들 것:

- 전체 레이아웃
- 헤더
- 메인 영역
- 기본 빈 상태 화면
- 공통 스타일 기준

완료 기준:

- `npm run dev`로 첫 화면을 볼 수 있습니다.
- 화면에 서비스 이름과 기본 설명이 보입니다.
- 모바일 화면에서도 내용이 잘리지 않습니다.

아직 하지 않을 것:

- 실제 API 연결
- 로그인
- 메모 저장
- 복잡한 라우팅

### 2. 기사 목록 화면

목표: 사용자가 경제 기사 목록을 훑어볼 수 있게 만듭니다.

만들 것:

- `ArticleListPage`
- `ArticleCard`
- 임시 기사 데이터
- 기사 제목, 요약, 출처, 발행일 표시

완료 기준:

- 여러 개의 기사 카드가 목록으로 보입니다.
- 카드 하나가 너무 많은 역할을 하지 않습니다.
- 임시 데이터 타입이 명확히 정의되어 있습니다.

아직 하지 않을 것:

- 서버 데이터 요청
- 페이지네이션
- 검색
- 필터

### 3. 기사 상세 화면

목표: 사용자가 기사 하나를 선택해 요약과 원문 링크를 볼 수 있게 만듭니다.

만들 것:

- `ArticleDetailPage`
- `/articles/:articleId` 라우트
- 목록에서 상세 화면으로 이동
- 원문 기사 링크

완료 기준:

- 기사 목록에서 카드를 누르면 상세 화면으로 이동합니다.
- 상세 화면에서 제목, 요약, 원문 링크를 확인할 수 있습니다.
- 존재하지 않는 기사 id에 대한 안내 화면이 있습니다.

아직 하지 않을 것:

- 메모 작성
- 읽음 완료 처리
- 인증이 필요한 기능

### 4. API 연결

목표: 임시 데이터 대신 백엔드 Article API에서 데이터를 가져옵니다.

만들 것:

- `api/articleApi.ts`
- `types/article.ts`
- API 기본 URL 환경 변수
- 목록 조회 API 연결
- 상세 조회 API 연결
- 로딩, 에러, 빈 상태

완료 기준:

- 서버가 켜져 있을 때 실제 기사 데이터가 보입니다.
- 서버가 꺼져 있거나 오류가 나면 에러 상태가 보입니다.
- 데이터가 없을 때 빈 상태가 보입니다.
- API 주소가 코드에 하드코딩되어 있지 않습니다.

아직 하지 않을 것:

- 클라이언트 캐싱 최적화
- 무한 스크롤
- 복잡한 에러 복구

### 5. TanStack Query 적용

목표: 서버 데이터의 로딩, 에러, 캐시 상태를 일관되게 관리합니다.

만들 것:

- QueryClient 설정
- 기사 목록 query
- 기사 상세 query
- query key 규칙

완료 기준:

- 직접 만든 `useEffect` 데이터 요청을 줄입니다.
- 로딩과 에러 처리가 query 상태를 기준으로 동작합니다.
- 같은 데이터를 다시 볼 때 불필요한 요청이 줄어듭니다.

아직 하지 않을 것:

- 낙관적 업데이트
- 복잡한 캐시 무효화

### 6. 개인 메모 작성

목표: 사용자가 기사를 읽고 자신의 생각을 남길 수 있게 만듭니다.

만들 것:

- 기사 상세 화면의 메모 입력 영역
- 메모 저장 버튼
- 메모 조회
- 메모 수정
- 저장 중 상태
- 저장 실패 상태

완료 기준:

- 메모를 작성하고 저장할 수 있습니다.
- 이미 저장한 메모가 다시 보입니다.
- 저장 버튼의 이벤트 핸들러 이름이 `handle`로 시작합니다.
- 빈 메모나 너무 긴 메모에 대한 최소한의 처리가 있습니다.

아직 하지 않을 것:

- 여러 개의 메모
- 리치 텍스트 에디터
- 자동 저장

### 7. 읽음 완료 처리

목표: 사용자가 오늘 읽은 기사를 완료 처리할 수 있게 만듭니다.

만들 것:

- 기사 상세 화면의 완료 버튼
- 완료 여부 표시
- 목록 화면의 완료 상태 표시
- 완료/취소 API 연결

완료 기준:

- 완료한 기사는 목록에서도 구분됩니다.
- 완료 버튼을 여러 번 눌러도 화면 상태가 이상해지지 않습니다.
- 실패 시 사용자에게 실패 상태가 보입니다.

아직 하지 않을 것:

- 통계 화면
- 연속 학습일 계산

### 8. 데일리 트래킹 화면

목표: 사용자가 오늘의 기사 읽기 진행 상황을 확인할 수 있게 만듭니다.

만들 것:

- `DailyTrackingPage`
- 오늘 읽은 기사 수
- 오늘 목표 수
- 완료한 기사 목록
- 아직 읽지 않은 기사 목록으로 이동하는 링크

완료 기준:

- 오늘의 진행률을 한눈에 볼 수 있습니다.
- 빈 상태와 로딩 상태가 있습니다.
- 숫자 표시가 서버 데이터와 맞습니다.

아직 하지 않을 것:

- 달력 UI
- 주간/월간 통계
- 알림

### 9. 라우팅과 내비게이션 정리

목표: 사용자가 앱 안에서 길을 잃지 않게 만듭니다.

만들 것:

- `routes/AppRoutes.tsx`
- `/articles`
- `/articles/:articleId`
- `/daily`
- 공통 헤더 내비게이션
- 404 페이지

완료 기준:

- 직접 URL로 접근해도 화면이 정상 표시됩니다.
- 없는 경로로 들어가면 안내 화면이 보입니다.
- 현재 위치를 알 수 있는 내비게이션이 있습니다.

### 10. 인증 화면

목표: 로그인 이후 개인 메모와 트래킹을 사용자별로 관리할 수 있게 준비합니다.

만들 것:

- 로그인 화면
- 회원가입 화면
- JWT 저장 방식 결정
- 로그인 상태에 따른 라우트 보호
- 로그아웃 버튼

완료 기준:

- 로그인하지 않은 사용자는 보호된 화면에 접근할 수 없습니다.
- 로그인 실패 메시지가 보입니다.
- 토큰이나 비밀값을 코드에 하드코딩하지 않습니다.

주의:

- 인증은 초반에 만들지 않습니다.
- 기사 목록과 상세 화면이 먼저 안정된 뒤 추가합니다.

### 11. 화면 다듬기

목표: 기능이 동작하는 상태에서 사용성을 개선합니다.

만들 것:

- 반응형 레이아웃 정리
- 버튼, 링크, 입력창 스타일 통일
- 로딩 UI 개선
- 에러 문구 개선
- 빈 상태 문구 개선

완료 기준:

- 모바일과 데스크톱에서 주요 화면이 깨지지 않습니다.
- 버튼과 링크의 역할이 명확합니다.
- 사용자가 다음에 무엇을 할 수 있는지 알 수 있습니다.

### 12. 배포 준비

목표: 프론트엔드를 운영 환경에 올릴 수 있게 준비합니다.

만들 것:

- 운영용 환경 변수
- 빌드 확인
- API 서버 URL 연결 확인
- 배포 후 주요 화면 확인

완료 기준:

- `npm run build`가 성공합니다.
- 운영 환경에서 기사 목록과 상세 화면이 열립니다.
- 환경 변수 없이 동작해야 하는 값과 환경 변수로 주입해야 하는 값이 구분되어 있습니다.

## 기능 작업 기본 순서

프론트엔드 기능은 아래 순서로 작게 구현합니다.

1. 필요한 타입을 정의합니다.
2. API 함수가 필요하면 `api` 폴더에 추가합니다.
3. 페이지 컴포넌트를 만듭니다.
4. 반복되는 UI를 컴포넌트로 분리합니다.
5. 로딩, 에러, 빈 상태를 추가합니다.
6. 모바일 화면을 확인합니다.
7. `npm run lint`와 `npm run build`를 실행합니다.

- 같은 코드가 세 번 이상 반복되고 있지는 않은가?

## 커밋 전 확인

`client` 폴더에서 아래 명령어를 실행합니다.

```sh
npm run lint
npm run build
```

오류가 나면 메시지를 읽고, 가장 위에 나온 오류부터 해결합니다.
경고는 무시하기 전에 왜 발생했는지 확인합니다.
